# 开发文档

## 本轮开发需求与结果总结

本轮目标是把原先“能跑的个人网站骨架”升级为“适合公开展示和持续更新的第一版成品”，重点不在于引入复杂动态系统，而在于完善内容展示、内容发现、公开发布能力和后续维护流程。实现上保留 `Astro + TypeScript + Markdown + content collections` 的静态架构，完成了首页与关于页的品牌化重构、博客与项目详情页的信息增强、博客精选区与项目精选区、文章归档、标签聚合、上一篇/下一篇与相关文章、RSS 输出、sitemap 与 robots 接入，以及基于环境变量的正式站点地址配置方式。当前版本已经通过 `npm run check`、`npm run lint` 与 `npm run build`，并成功生成归档页、标签页、RSS 与 sitemap。

## 本轮新增与升级的关键内容

- `src/data/site.ts`
  升级为全站品牌文案中心，统一首页主文案、当前关注方向、联系引导、关于页画像信息和发布检查清单。

- `src/content.config.ts`
  博客新增 `featured`、`updatedAt`；项目新增 `role`、`status`、`period`、`highlights`，为展示升级和内容发现提供字段支撑。

- `src/pages/index.astro`
  重构为更强调个人品牌表达的 editorial 首页，包含主视觉、当前关注、精选文章、精选项目、最近更新和联系入口。

- `src/pages/about.astro`
  从简单信息页升级为个人画像页，加入简介、能力地图、学习方向和外部链接。

- `src/pages/blog/[slug].astro`
  增加更新时间、上一篇/下一篇与相关文章，提升阅读连续性。

- `src/pages/projects/[slug].astro`
  增加角色、状态、项目周期与亮点信息，使项目详情更接近可复盘的案例页。

- `src/pages/archive.astro`
  新增按年份浏览文章的归档页。

- `src/pages/tags/[tag].astro`
  新增按标签自动聚合的标签页。

- `src/pages/rss.xml.js`
  新增 RSS 输出，便于订阅博客更新。

- `astro.config.mjs`
  接入 `@astrojs/sitemap`，并通过 `PUBLIC_SITE_URL` 支持正式域名下的 canonical、RSS 与 sitemap。

- `src/pages/robots.txt.ts`
  新增 robots 输出。

## 当前发布与部署约定

- 正式域名不写死在业务文案中，统一通过 `PUBLIC_SITE_URL` 环境变量提供。
- 本地仍使用 `127.0.0.1:3000` 进行开发。
- 发布前必须执行：
  - `npm run check`
  - `npm run lint`
  - `npm run build`
- 人工预览重点页面：
  - 首页
  - 文章详情页
  - 项目详情页
  - 归档页
  - 标签页

## 当前遗留与后续建议

- `public/og-default.svg` 目前只是占位资源，后续应替换为真正的默认社交分享图。
- `src/content.config.ts` 中 `z.string().url()` 产生的弃用提示暂不阻塞功能，但后续升级 Astro / Zod 时应统一清理。
- 如果正式部署时浏览器页面仍出现中文乱码，再把编码治理提升为阻塞问题单独处理。

## 2026-05-11 Vercel 上线准备

本轮目标是把项目从“位于上层仓库中的站点目录”推进到“适合独立托管与部署的仓库形态”，并补齐上线前的最小清理工作。实现上移除了内容模型中暂未接入渲染链路的 `cover` 字段，避免 schema 与页面能力不一致；把 `portfolio-site` 的仓库地址改为预期真实仓库地址，并删除 `study-tracker` 中并不存在的演示链接，避免线上访客点击到占位页面；同时在 `README.md` 中补充了 `DEFAULT_SITE_URL` 与 `PUBLIC_SITE_URL` 的职责边界，以及根域名与 `www` 的推荐配置方式。另在 `.gitignore` 中加入 `.vercel/`，为后续接入 Vercel 保持仓库整洁。默认分享图也从二进制占位 `png` 调整为文本可维护的 `svg`，并取消了对 `favicon.ico` 的依赖，方便同步部署关键文件。

## 2026-05-12 网站部署知识沉淀

本轮目标不是继续改站点功能，而是把“已经真实走过一次的上线流程”沉淀成适合后续复习的入门资料。新增了根目录文档 `网站部署与DNS快速入门.md`，围绕 GitHub、Vercel、域名、DNSPod、HTTPS 与 CDN 的关系做了面向初学者的解释，并结合本项目的真实上线链路整理了 DNS 记录、域名验证、证书签发和常见报错排查。文档中加入了 Mermaid 流程图、时序图和按难度分层的阅读链接，便于快速建立整体地图，而不是零散记术语。

## 2026-05-12 首页与栏目重构

本轮目标是把网站从“内容很多但首页偏拥挤”的状态，调整为“首页负责介绍与导流，栏目页负责承载内容”的更清晰结构。实现上没有引入复杂分类系统，而是给博客内容新增显式 `category` 字段，仅划分为 `生活感悟` 与 `学习笔记` 两类，项目继续保留为独立的 `个人项目` 集合；首页则删除整块的精选文章与精选项目展示，只保留站点简介、三个板块入口和最近更新，从而降低首次访问的认知负担。同时新增 `/life` 与 `/study` 两个栏目页，保留原有文章详情链接不变，减少历史路径失效风险，并在文章卡片与详情页中补上栏目语义，使整站的信息架构更贴合真实内容意图。

## 2026-05-12 视觉风格规划

本轮目标不是马上改代码，而是先把网站的视觉升级方向拆成两套可执行规划：一套参考 P3R 的天蓝白天模式，一套参考 P5R 的红黑黑夜模式。已经分别新增 `P3R白天模式规划.md` 与 `P5R黑夜模式规划.md`，每份文档都包含参考来源、风格关键词、配色、组件建议、Mermaid 结构图和落地边界，方便后续真正实现主题切换时直接按图施工，而不是重新从零讨论审美方向。

## 2026-05-12 主题切换机制与白天模式落地

本轮先完成了“机制先行”的主题系统，而不是直接堆两套平行样式。实现上在 `BaseLayout` 中提前注入主题恢复脚本，用 `data-theme` 驱动全站 token，配合 `ThemeToggle` 组件完成白天 / 黑夜切换与 `localStorage` 持久化，确保刷新后仍能保持用户选择，同时避免首屏闪屏。视觉上先只正式落地 P3R 方向的白天模式：全站从原来的暖米色调整为冷蓝白、透明叠层、柔和高光、玻璃感卡片和更现代的导航/按钮语言；黑夜模式目前仅建立 token 与切换底座，后续再按 P5R 规划继续把红黑高对比海报感完整做出来。

## 2026-05-12 白天模式二次修正

本轮针对“白天模式虽然变蓝，但仍然偏白、偏普通网页”的问题做了第二次视觉修正。调整重点不再只是 token，而是把首页构图一并拉向更接近 P3R 的方向：主色从浅雾蓝推进到更有力量的海蓝 / 群青，同时引入黑色大字、白色切片、青色高光和圆形蓝面，让首页从常规博客式排版转向更具海报感的作品首页。实现上在首页 Hero 中加入编号、夸张背景字、斜切标签和大面积蓝色圆形结构，三个入口卡片也增加轻微错位与编号感；与此同时保留正文页的克制程度，保证“首页更有艺术张力，内容页仍然易读”。

## 2026-05-12 白天模式局部精修

本轮在继续强化白天模式设计感时，优先修复了 Hero 区域的遮挡问题，并进一步精修了两个局部：导航/按钮/标签，以及首页三个入口卡。处理方式不是简单删除装饰，而是把正文、说明文字、按钮和眉标题统一抬到更高的层级，让装饰字与蓝色块面退回背景层；与此同时把入口卡改成更接近功能面板的切角样式，并给主按钮、次按钮和状态标签加上更明确的 P3R 式斜切节奏。这样既保住了海报感，也避免“设计元素盖住信息本体”的问题再次出现。
## 2026-05-12 黑夜模式正式落地

本轮目标是在既有主题切换机制上，把参考 P5R 的黑夜模式真正做成完整可用的界面，而不是只把白天模式的蓝色替换成红色。实现上坚持“共享结构，夜间覆盖”的策略：不复制页面和组件，而是在 `html[data-theme='night']` 下，为首页 Hero、导航、按钮、栏目入口卡、最近更新卡、文章详情页、项目详情页和页脚分别补夜间版视觉规则。首页被处理成更强的红黑海报感，采用黑底、红色楔形块面、白字高对比和斜切标签；三个入口卡与最近更新区则统一成更像“任务卡 / 行动记录”的夜间语言。与白天模式阶段不同，这次显式控制了装饰层级：所有标题、正文、按钮和说明文字都保持在前景层，红黑装饰块、编号和背景字严格后退到背景层，从实现上规避了遮挡问题。验证方面完成了 `npm run check`、`npm run build`，并通过本地预览加 Chrome DevTools 协议截图检查了夜间首页、文章页和项目页，确认切换、持久化、可读性和层级都正常。
